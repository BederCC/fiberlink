Informe Técnico del Sistema

Análisis completo de la arquitectura del portal de gestión de ISP FiberLink, incluyendo la estructura de directorios, el modelo de base de datos relacional y los flujos de optimización de facturación masiva.

Tecnologías e Idioma

Servidor (Backend)

PHP 8.0+

Toda la lógica de negocio, manejo de base de datos relacional (PDO), envío de correos automatizados (SMTP/PHPMailer) y generación de reportes en PDF (TCPDF) está escrita enteramente en **PHP puro (Vanilla PHP)**.

Diseño (Frontend)

Tailwind CSS & JS

La interfaz de usuario es responsiva, moderna y limpia. Utiliza clases utilitarias de **Tailwind CSS** (vía CDN o local) junto con **JavaScript nativo (ES6)** para solicitudes asíncronas dinámicas (Fetch API) y animaciones de control.

Base de datos

MariaDB / MySQL

Los datos persistentes de clientes, facturas, productos, servicios y registros de actividad del portal se gestionan en un motor relacional de base de datos **MySQL** optimizado con índices y claves foráneas.

Arquitectura del Sistema

Modelo Cliente-Servidor Desacoplado

La aplicación implementa una arquitectura **Cliente-Servidor clásica**. El servidor expone puntos de acceso estructurados en formato **JSON** a través de archivos controladores API en la carpeta /api. El cliente (el navegador) carga las páginas HTML estáticas en la carpeta /public, interactúa con el usuario y se comunica con el servidor a través de peticiones HTTP asíncronas asiladas empleando JavaScript nativo (fetch()).

El Servidor (PHP + MySQL)
  • PDO (PHP Data Objects): Proporciona una capa segura contra inyecciones SQL usando consultas preparadas.
  • Servicios REST-like: APIs ligeras que capturan cabeceras personalizadas y retornan respuestas con códigos de estado HTTP (200, 201, 400, 401, 503).
  • Módulo de PDF & SMTP: Generación de facturas físicas con TCPDF y envío por SMTP de Hostinger.
El Cliente (Navegador)
  • Renderizado Dinámico: Las tablas de la interfaz se cargan dinámicamente mediante consultas a las APIs locales, reduciendo el tráfico y mejorando el tiempo de respuesta.
  • Barra de Progreso y Batching: Segmentación de grandes tareas en lotes controlados para no saturar las capacidades del servidor del hosting compartido.
  • Persistencia de Sesión: Control mediante sesiones de servidor estándar (session_start()).

Estructura de Base de Datos

El sistema relacional consta de las siguientes tablas estructuradas:

Tabla Descripción Claves Importantes / Relaciones
clients Mantiene la información personal de los abonados (DNI/RUC, teléfono, dirección, coordenadas GPS). PK: id (Auto-incremental) | UNIQUE: dni_ruc
services Registra los servicios asignados a cada cliente, detallando IP, dirección MAC física, modelo del router y estado de conexión. FK: client_id (hacia clients.id), FK: plan_id (hacia plans.id)
plans Especifica el catálogo de velocidades (Mbps) y precios de planes de fibra óptica mensual. PK: id
installations Bitácora de campo que asigna instalaciones a técnicos y registra costos de mano de obra y estado de las obras. FK: client_id (hacia clients), FK: technician_id (hacia users)
installation_details Tabla intermedia para registrar los equipos y accesorios consumidos en una instalación. FK: service_id (hacia services), FK: product_id (hacia products)
products Catálogo de equipos y materiales en stock (Routers, repetidores, cables de fibra drop, conectores). PK: id
invoices Facturas comerciales generadas de manera recurrente (mensuales) o de instalación única. FK: client_id (hacia clients) | UNIQUE: invoice_number
invoice_items Detalle de conceptos facturados por cada recibo comercial generado. FK: invoice_id (hacia invoices) (ON DELETE CASCADE)
payments Registros de cobros procesados asociados a facturas específicas con su forma de pago y ID de operación. FK: invoice_id (hacia invoices), FK: created_by (hacia users)
payment_metrics Control de tiempo transcurrido desde la consulta hasta la realización del pago. FK: invoice_id (hacia invoices)
users Control de accesos y credenciales con roles definidos (admin, técnico, ventas). PK: id | UNIQUE: username

Flujos Claves y Optimizaciones

Doble Entorno Dinámico Auto-Detectado

Para evitar que al subir cambios del entorno local se rompa la conexión de producción, la clase Database en config/database.php analiza las variables de entorno del servidor. Si el sistema operativo es Windows (XAMPP) o el dominio es local (ej. localhost, 127.0.0.1, .local), el script se conecta automáticamente con la base de datos de desarrollo. Si se ejecuta en Linux (Hostinger), se conecta usando las credenciales del host web, evitando modificaciones de código repetitivas.

Facturación Masiva Segura (Batching & Retries)

Generar cientos de facturas en una sola llamada satura los límites de tiempo de ejecución del servidor. En public/billing.php, el cliente procesa la generación por lotes de **5 facturas a la vez**. Entre cada lote se aplica una pausa de **500 ms** para liberar bloqueos. Adicionalmente, cuenta con un sistema de **reintento automático** (hasta 2 intentos con 1.5s de espera ante fallos temporales), visualizando el proceso en una barra de progreso que indica el porcentaje y estado exacto en tiempo real.

Automatización de Pago y Envío de Comprobantes

Al registrarse un pago total sobre una factura en api/payments.php, el sistema automáticamente: 1. Cambia el estado de la factura a 'paid'.
2. Identifica si el cliente poseía servicios 'suspended' (suspendidos) y los reactiva a 'active'.
3. Invoca la clase PdfGenerator para estructurar el PDF con TCPDF.
4. Utiliza Mailer para notificar al cliente vía email adjuntando el comprobante en formato PDF.

Guía de Instalación y Pruebas Local

1 Herramientas y Requisitos Previos

Antes de comenzar, asegúrate de descargar e instalar los siguientes programas en tu computadora:

XAMPP

Servidor Apache, PHP 8.0+ y base de datos MariaDB/MySQL.

VS Code

Editor de código recomendado para editar configuraciones.

Node.js

Requerido para gestionar paquetes y compilar Tailwind CSS.

Composer

Gestor de dependencias de PHP para las utilidades de correo y PDF.

2 Pasos de Instalación y Despliegue

Paso A: Ubicación en htdocs

Descarga o clona el repositorio desde GitHub y mueve la carpeta completa de fiberlink al directorio de publicación web de XAMPP: 

Windows:  C:\\xampp\\htdocs\\fiberlink
Linux:    /opt/lampp/htdocs/fiberlink
Paso B: Importación de Base de Datos
  1. Inicia el panel de control de XAMPP y activa los módulos Apache y MySQL.
  2. Ingresa a tu navegador y entra a phpMyAdmin: http://localhost/phpmyadmin/.
  3. Crea una nueva base de datos llamada fiberlink (con cotejamiento utf8mb4_general_ci).
  4. Haz clic sobre tu nueva base de datos, ve a la pestaña de Importar, selecciona el archivo database.sql (ubicado en la raíz del proyecto) y haz clic en Importar / Continuar.
Paso C: Configuración de Archivos de Entorno

En la carpeta config/ del proyecto, copia y renombra los archivos de ejemplo:

  • Base de Datos: Duplica config/database.php.example y renombralo como config/database.php. El archivo viene configurado por defecto con el usuario root y contraseña vacía, ideal para el entorno local predeterminado de XAMPP.
  • Correo Electrónico (Notificaciones SMTP): Duplica config/mail.php.example y renombralo como config/mail.php. Abre el archivo e introduce tus credenciales de correo (Servidor SMTP, email y contraseña) si deseas probar la funcionalidad de envío de recibos y facturas en formato PDF por email.
Paso D: Instalación de Dependencias

Abre la consola o terminal del sistema, dirígete a la raíz del proyecto (cd C:\\xampp\\htdocs\\fiberlink) y ejecuta los siguientes comandos:

1. Instalar dependencias de PHP (PHPMailer, TCPDF): 
composer install
2. Instalar paquetes de Node (Tailwind CSS compiler): 
npm install
3. Compilar / Monitorear Tailwind CSS en desarrollo local: 
npx tailwindcss -i ./src/input.css -o ./public/dist/output.css --watch
Paso E: Ejecución e Ingreso al Panel

Abre tu navegador de preferencia e introduce la siguiente dirección local: 

http://localhost/fiberlink/public/login.php

Credenciales de Administrador por defecto registradas en la base de datos:

  • Usuario: admin
  • Contraseña: admin

Despliegue Continuo en Hostinger

1 Integración Git / GitHub en Hostinger

Hostinger permite la automatización de despliegues (CI/CD) de modo que cada vez que haces un push (git push) desde tu editor (ej. VS Code) a GitHub, los cambios se reflejan automáticamente en el servidor web de Hostinger.

Paso A: Configuración en Hostinger hPanel
  1. Inicia sesión en tu cuenta de Hostinger y dirígete al panel de tu sitio web (hPanel).
  2. En la barra lateral izquierda, navega a Avanzado -> Git.
  3. En la sección "Desplegar repositorio Git", introduce los siguientes datos:
    • Dirección del Repositorio: https://github.com/BederCC/fiberlink.git
    • Rama: main
    • Directorio de instalación: public_html (o la subcarpeta donde alojes la web).
  4. Haz clic en el botón Crear.
Paso B: Webhook de Automatización Real-Time
  1. Una vez creado el repositorio Git en Hostinger, verás un enlace llamado "Webhook URL". Cópialo.
  2. Entra a tu repositorio en GitHub y haz clic en la pestaña Settings (Configuración).
  3. En el menú de la izquierda, entra a Webhooks y haz clic en Add webhook (Agregar webhook).
  4. Pega la URL copiada en el campo Payload URL y asegúrate de seleccionar:
    • Content type: application/json
    • Events: "Just the push event." (Solo el evento push).
  5. Guarda el webhook. ¡Listo! A partir de ahora, cada git push actualizará tu web en producción de inmediato.

2 Configuración de Base de Datos en Producción

La base de datos en Hostinger es independiente del entorno local. Para mantener la consistencia:

Importar Esquema SQL

Entra a la sección de Bases de Datos -> Bases de Datos MySQL en el panel de Hostinger, crea una base de datos e importa el archivo database.sql de tu repositorio usando phpMyAdmin en la nube.

Modificar Credenciales

Al crear la base de datos MySQL en Hostinger, el sistema generará un nombre de base de datos y de usuario con un prefijo (ej. u226387153_fiberlink).

Abre tu archivo local config/database.php y asegúrate de rellenar el bloque de producción else con esas credenciales exactas antes de hacer push.
